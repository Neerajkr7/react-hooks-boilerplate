AI Concepts: RAG, MCP, Agents & Workflows
Team training session — foundational concepts
01

What is an LLM?

An LLM (Large Language Model) is a model trained on a broad corpus of information. It understands general concepts — what a car looks like, that the sky is blue — but has no knowledge of your specific domain or private data.


Example: An LLM may know what a "product" or a "SKU" is in general terms, but it will not know what a catalog means in the context of Equinox, or the difference between a Sales Catalog and a Master Catalog, or any collection properties specific to your platform.

02

RAG — Retrieval-Augmented Generation

Definition

R Retrieval  A Augmentation  G Generation


When the LLM receives a query, it retrieves relevant data from an external source, augments (adds) that data to the input, and then generates a response. The retrieved data is called context.

Context injected into LLM

Output generated
03

Knowledge Base

What it is

The Knowledge Base is an external hard drive for the LLM — a dedicated store of domain-specific information the model can retrieve from, beyond what it was originally trained on.

How it's stored

  • All uploaded documents are converted into vector embeddings and stored in a vector store.
  • During a chat session, retrieval pulls the relevant embedded chunks and provides them back to the LLM as context.
  • Each agent or service can have its own dedicated knowledge base (e.g., Catalog Knowledge Base with developer portal documentation).

Maintenance note: Currently, documentation is manually pushed to the knowledge base. A future automation pipeline is planned to sync updates from the developer portal directly.

04

MCP — Model Context Protocol

Purpose

MCP is a middleware layer between your existing APIs and the LLM. Rather than connecting the LLM directly to raw APIs (which creates complexity), MCP wraps each service's APIs into structured tools the LLM can understand and invoke.

Structure

  • Each service (e.g., SFO, Catalog, Price) gets its own MCP.
  • Each API endpoint within that service is represented as a Tool.
  • Each tool carries metadata: what data the API needs, expected request body, and response format.

How LLM uses it

User question
→
LLM identifies best tool
→
LLM builds API request
→
API is called
→
Response returned

Example: In SFO MCP, the Load Products by Search Term API is represented as a search tool. The LLM picks this tool when a search-related question arrives and auto-constructs the request.

Key insight: MCP is generated by the Equinox MCP Generator and deployed as a separate service. Direct API-to-LLM integration is avoided because it is too complex and error-prone.

What is an Agent?

Definition

Something is only called an Agent if it has all five of the following elements:

①
LLM
The model/brain
②
Memory
Short & long term
③
Knowledge
Optional knowledge base
④
Prompt
System instructions
⑤
Tools
MCPs / APIs

Memory Types

  • Short-term memory: Within the current chat thread only.
  • Long-term memory: Persisted facts about a user across sessions (e.g., "this user is an IT professional attending a meeting").

Tool Scope = Agent Scope

An agent can only do what its assigned tools allow. Give it catalog tools → catalog agent. Add pricing tools → it handles pricing too. Tools define the domain boundary.

⚠️ Too many tools = worse performance. Every tool consumes tokens from the LLM's context window (~200k tokens for most models). Overloading an agent with tools leaves less context for actual reasoning, causing hallucinations. Keep agent tool sets focused and domain-specific.

06

Guardrails

What they are

Guardrails are a screening layer placed both before and after the LLM — preventing harmful, inappropriate, or sensitive content from flowing in either direction.

Two layers of guardrails

  • Input guardrail: Checks what the user sends to the LLM. Blocks harmful prompts (e.g., instructions for illegal activity, inappropriate content).
  • Output guardrail: Checks what the LLM returns before it reaches the user. Masks or blocks PII — phone numbers, email IDs, personal data that should not be exposed.

→
LLM
→

07

Workflows

What they are

Workflows are AI Process Automation — a sequence of steps defined upfront that the system executes automatically. Unlike chat, you don't need to manually prompt each step; the workflow knows what to do.

Three types of workflows

  • Chat-based: Initiated and driven by a conversation. User chats with the workflow to generate output.
  • Trigger-based: Fires automatically when a specific event occurs (e.g., a product is updated in the catalog).
  • Schedule-based: Runs on a defined schedule (e.g., generate a product review report every night at 9 PM IST).

Workflow Creation — Multi-Agent Builder

Creating a workflow via the builder involves a supervisor agent coordinating four sub-agents:

  • Resource Explorer: Scans available agents, knowledge bases, MCPs, and tools.
  • Workflow Designer: Designs the workflow as a JSON structure based on the supervisor's findings.
  • Workflow Validator: Validates the generated JSON before activation.
  • Workflow Tester: Runs a test round before activation (changes cannot be made post-activation).
08

Orchestrator / Supervisor Agent

Pattern

For complex domains (like Admin Assistant), a single agent with all tools would exceed the context limit. Instead, an Orchestrator Agent sits on top and routes requests to specialized sub-agents.

Example — Admin Orchestrator

  • Orchestrator receives any user input.
  • It identifies the right sub-agent (Catalog, Price, Inventory, etc.).
  • Delegates the task, gets the result, and presents the final answer.
  • Sub-agents stay focused — Catalog agent has ~30 catalog tools; Price agent has ~10 price tools.

Agent-to-Agent (Future)

The platform is working toward Agent-to-Agent communication — where one agent can hand off to an entirely separate external agent (e.g., integrating with SendGrid directly via its MCP).

01

Macy's Demo — Overview

What it is

A standalone AI-powered fashion chatbot built as a proof-of-concept using scraped Macy's product data. Repo: EQ Platform Macy's Demo

Data

  • ~12,000 products and ~93,000 SKUs scraped from Macy's website.
  • All 12,000 products converted into vector embeddings stored in a JSON file (no database — JSON was manageable at this scale).
  • Agent queries these indexes using cosine similarity to find relevant products.

Limitation: Macy's products have 30–50 SKUs per product. Equinox platform currently has a product-SKU combination limit, so full integration with Equinox store is blocked. Workaround: limit SKUs per product for new demos.

Key Design Decisions in the Repo

  • Ask only ONE clarifying question — multiple questions loop endlessly without producing results.
  • Category-aware discovery: AI queries across men's/women's categories, accessories, belts, wallets, shoes and assembles a complete outfit.
  • Discovery uses direct embedding query — not a tool call. Reads directly from the index, faster than going through an MCP tool.
  • Full agent instructions/prompt defined inside the Macy's demo repo.

Shopping MCP — Concept & Architecture

What it is

A custom, handwritten MCP server purpose-built for the end-to-end shopping journey. Connects any LLM (Claude, ChatGPT, VS Code Copilot) directly to Equinox storefront APIs.

APIs included (~10 handpicked)

  • Product search
  • Add item to cart
  • Get customer address
  • Get customer payment details
  • Checkout

Each API = one MCP Tool. LLM picks the right tool and constructs the API request itself.

MCP Apps (UI Layer)

An additional UI layer bundled inside the MCP server. Instead of tabular/text output, tool responses render as visual product cards. This is the standard MCP Apps feature for visual output from tool calls.

Shopping Journey — B2C vs B2B

B2C
  • Search product
  • Get details
  • Add to cart
  • Checkout
B2B
  • AI recommends items not ordered recently
  • "You might want to restock X"
  • Add recommended items to cart
  • Checkout with PO number

Status: Built in TypeScript. Not yet deployed. Most code is done — deployment + minor optimizations remain.

Pending optimization: Merge separate checkout tools (get address, get payment, checkout) into a single tool that accepts a B2B/B2C flag and handles the full flow internally.

05

Storefront MCP vs Dedicated Shopping MCP

Current: Storefront MCP + Agent Swarm

  • Uses the generic Storefront MCP which has ~200 tools.
  • Separate agents per domain: checkout agent, discovery agent, etc.
  • All sub-agents report to a Shopping Assistant agent.
  • Problem: High latency — round-trip communication between multiple agents adds delay.

Proposed: Dedicated Shopping MCP

  • Single MCP server with only the ~10 APIs needed for the shopping journey.
  • One agent talks directly to this MCP — no multi-hop agent routing.
  • Benefit: Faster responses, simpler architecture, less hallucination risk.

Fallback Plan

If Shopping MCP deployment is delayed, the Macy's repo can be adapted to work as an agent using existing MCPs. Discovery (embedding-based) and outfit curation are the only unique capabilities Macy's has over direct MCP usage.

06

Tech Stack & Deployment

Current Stack

  • Language: TypeScript (both Macy's demo and Shopping MCP)
  • No LangChain / LangGraph — not using the AI Studio framework here
  • MCP Apps UI: HTML + CSS (must stay TypeScript since UI is bundled)
  • Deployment: Standalone Node app — currently on a repurposed node/port; needs proper DevOps setup

Python Conversion?

Converting Shopping MCP to Python is possible, but MCP Apps (the UI layer) requires HTML/CSS, which ties it to TypeScript. Demo/POC work can continue in TypeScript; production must align to the standard Equinox stack.

Productization rule: When moving from demo to production, everything must align to the standard Equinox tech stack. No multiple tech stacks in production.
